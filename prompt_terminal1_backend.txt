AUFGABE 2 — HOOK-SYSTEM ECHTE IMPLEMENTIERUNG (Priorität: HOCH)
================================================================================

Ersetze den STUB in backend/src/core/hooks/hook-runner.ts mit einer echten Implementierung:

Die echte Implementierung muss:
1. hook_definitions aus der DB laden (WHERE tenant_id = ? AND event_name = ? AND is_active = true)
2. Für jeden aktiven Hook: HTTP POST an endpoint_url senden mit Payload { event, receipt, timestamp }
3. HMAC-SHA256 Signatur im Header X-ProzessPilot-Signature (secret aus hook_definitions)
4. Timeout: timeout_ms aus DB (default 10000ms) — nutze AbortController
5. Retry: bis zu retry_count Versuche bei Netzwerkfehler (exponentielles Backoff: 1s, 2s, 4s)
6. Execution in hook_executions Tabelle loggen (immer, egal ob Erfolg oder Fehler)
7. Bei Hook-Fehler: Receipt NICHT stoppen — Fehler nur loggen, Pipeline läuft weiter
8. Unterstützte Events: before_extraction, after_extraction, before_categorization, after_categorization, after_archiving, receipt_completed, receipt_error

Erstelle auch backend/src/core/hooks/hook.repository.ts mit:
- getActiveHooks(tenantId, eventName) → HookDefinition[]
- logExecution(data) → void
- createHook(tenantId, data) → HookDefinition
- updateHook(tenantId, hookId, data) → HookDefinition
- deleteHook(tenantId, hookId) → void

Erstelle backend/src/core/hooks/hook.routes.ts:
- GET /hooks → alle Hooks des Tenants
- POST /hooks → Hook erstellen
- PUT /hooks/:hookId → Hook bearbeiten
- DELETE /hooks/:hookId → Hook löschen
- GET /hooks/:hookId/executions → Execution-Log der letzten 50

Registriere die Routes in backend/src/app.ts.

Schreibe Tests in backend/src/core/hooks/__tests__/hook-runner.test.ts:
- Hook mit Erfolg → execution geloggt, status 200
- Hook mit Timeout → execution geloggt als error, Pipeline läuft weiter
- Hook mit falschem Secret → wird trotzdem gesendet (Empfänger verifiziert)
- Kein Hook in DB → Receipt wird unverändert zurückgegeben

AUFGABE 3 — WF-ERROR-HANDLER n8n Workflow (Priorität: MITTEL)
================================================================================

Erstelle n8n/workflows/WF-ERROR-HANDLER.json als vollständigen n8n-Workflow.

Der Workflow:
1. Trigger: Webhook POST von WF-MASTER-RECEIPT bei Fehler (/webhook/error-handler)
2. Input: { receiptId, tenantId, customerId, stage, error, errorType }
3. Klassifiziert Fehler nach errorType:
   - OCR_FAILED → receipt status = requires_review, Benachrichtigung an Steuerberater
   - CATEGORY_FAILED → receipt status = requires_review, Confidence war zu niedrig
   - ARCHIVE_FAILED → receipt status = error, Retry-Mechanismus (3 Versuche)
   - WEBHOOK_FAILED → nur loggen, Pipeline läuft weiter
4. Setzt Receipt-Status via PUT /receipts/:id/status
5. Schreibt Fehler in error_log Tabelle via Backend-Endpoint POST /errors
6. Sendet WhatsApp-Benachrichtigung wenn customer WhatsApp aktiv hat (M07 nutzen)

Der Workflow muss das gleiche JSON-Format haben wie die anderen Workflows in n8n/workflows/.
