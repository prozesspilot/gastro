**Recovery-Button (Stack wieder hochfahren):**
```
gh workflow run deploy-staging.yml --ref main
```
Das SSHt mit dem CI-Key auf den Server und macht `git reset --hard origin/main` → pre-deploy-Backup → `docker compose -f docker-compose.prod.yml up -d` → `migrate:prod` → Caddy-Reload. **Achtung:** deployt damit IMMER den aktuellen `main` (inkl. neuer Migrations), nicht nur „neustarten".

**Zwei Deploy-Bugs in PR #138 (2026-06-17) gefixt**, die Recovery vorher unmöglich machten: (1) der `deploy`-Job hatte `if: github.event_name == 'push'` → manueller „Run workflow" baute nur Images, deployte nie; jetzt auch `workflow_dispatch`. (2) Pre-Deploy-`pg_dump` brach mit `set -e` ab, wenn Postgres unten war (Henne-Ei im Recovery-Fall); jetzt `pg_isready`-Guard → bei totem Postgres Backup überspringen + weiter. Reihenfolge: `up -d` (Stack kommt hoch) kommt VOR `migrate` → selbst wenn Migrationen haken, ist der Stack oben.
