**Verifiziert 2026-06-02** beim Audit nach dem T041-RLS-Fix (siehe [[rls-guc-key-mismatch]]). Mehrere Code-Pfade aus der **Legacy-`customer`/`receipt`-Welt** wurden beim Reboot auf `tenants`/`belege` (ADR-004 / T028) nie umgestellt und laufen jetzt gegen ein Schema, das ihre Spalten nicht hat — sie scheitern zur Laufzeit, aber **silent** (try/catch + DB-mockende Tests).

**Befund 1 — ✅ ERLEDIGT (T043, PR #227+#228, 2026-06-30):** `modules/tenants/` war seit dem A3-Reboot (T058: Live-`/tenants` läuft über `routes/tenants.routes` + `list_tenants_for_staff()` 121) verwaiste Hülle. Das tote Falsch-Spalten-CRUD (`createTenant`/`listTenants`/`findTenantById`/`updateTenant`, `name`/`active`) + `tenant.routes.ts` wurden gelöscht. Die EINZIG live genutzte Funktion `tenantExists` (M01-Upload) hatte einen echten latenten Prod-Bug: nackter `pool.query` ohne Tenant-Kontext → unter `gastro_app` (NOBYPASSRLS) gibt die FORCE-RLS-Policy 0 Zeilen → M01-Upload hätte in Prod jeden Beleg abgelehnt. Fix: SECURITY-DEFINER-Fn `tenant_exists(uuid)` (Migration 130, Muster 121) + gastro_app-Integrationstest. Offen bleibt nur: verwaister Legacy-Schema-Cluster `core/schemas/{tenant,customer,document,routing-job}.ts` (barrel-exportiert in `core/schemas/index.ts` + in `tests/schemas/schemas.test.ts` getestet, sonst referenzlos) → eigener Cleanup-Folge-Task.

**LEHREN (2026-06-30):** (1) **gastro_app-DB-Integrationstests müssen Tabellen-Grants selbst im `beforeAll` setzen** (`GRANT SELECT ON <tabelle> TO gastro_app`) — sonst wirft der Query `permission denied` (42501) BEVOR RLS greift, und der Test ist **test-reihenfolge-abhängig flaky** (grün nur, wenn ein anderer Test das Grant zufällig vorher setzte). CI seedet gastro_app NICHT mit Tabellen-Grants (das macht nur `setup-app-role.sql`, das CI nicht ausführt). (2) **NIE `gh pr merge --admin` bei rotem/pending Backend-CI** — genau das ist mir bei #227 passiert (flaky Test rot zum Merge-Zeitpunkt); Fix-Forward nötig (#228). Vor Admin-Merge IMMER `fail+pending == 0` bestätigen.

**Befund 2 — audit_log-Writer-Drift ✅ GRÖSSTENTEILS ERLEDIGT (verifiziert 2026-06-30, T042-Check):**
Ursprünglich: 8 Wrapper schrieben `(tenant_id, actor, action, resource, payload)` gegen das echte Schema (`entity_type, entity_id, event_type, actor, payload_before/after, metadata`). **Stand 2026-06-30:** Die Legacy-Module (m02/m07/m08-alt/m10, `_shared/receipts`) wurden im Reboot ENTFERNT; Business-Audit läuft jetzt korrekt über `core/audit/audit-log.ts::logAuditEvent` (12 Live-Nutzer: M01-Writer, M05 markBelegExported, M15 purge, confirmBelegReview, …) + `dsgvo/loeschung.service.ts` (korrekte Spalten). **Einziger verbliebener Falsch-Spalten-Writer:** `src/core/audit/audit.service.ts::log()` (`action/resource/payload`) — aber **toter Code, KEIN Importeur** → kein Laufzeit-/GoBD-Schaden. Die P1-Sorge „kein Business-Audit" trifft NICHT mehr zu. Rest-Aufgabe war nur noch das tote `audit.service.ts` löschen → ✅ **erledigt (PR #226, 2026-06-30)** (+ verwaister `tests/audit/audit.service.test.ts`). **Befund 2 vollständig aufgelöst.** T042 damit faktisch obsolet (nur noch der `tenant.repository`-Teil = Befund 1 / T043 offen).

**Warum unsichtbar:** Tests mocken die DB per `if (/INSERT INTO audit_log/i.test(sql)) return {rows:[]}` — Spaltennamen werden nie gegen das echte Schema validiert. Gleiche Klasse wie der RLS-GUC-Bug: nur unter der echten Prod-Konstellation (`gastro_app` + echtes Schema) sichtbar.

**Konsequenz:** Diese Befunde sind starke Argumente für **T028 Option A (Legacy-Welt abbauen)** statt reparieren. Falls Koexistenz: alle 8 Audit-Writer auf `logAuditEvent` (mit `setTenantContext`) umstellen + Real-DB-Test gegen `audit_log`, der Schema-Drift CI-rot macht. `tenant.repository.ts` auf neues Schema + `withTenant`/Bypass umstellen.
