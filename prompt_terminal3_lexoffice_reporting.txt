lexoffice.client.ts muss:
- Auth via API-Key (aus customer_credentials Tabelle, verschlüsselt) — kein OAuth2 nötig in Phase 2
- Basis-URL aus ENV: LEXOFFICE_API_BASE=https://api.lexoffice.io
- Timeout aus ENV: LEXOFFICE_DEFAULT_TIMEOUT_MS=15000
- Rate-Limiter: max. 2 Requests/Sekunde pro Mandant (Token-Bucket, in-memory für Phase 2, kein Redis)
- Retry: 3x exponential bei 5xx oder 429 (Retry-After-Header beachten)
- Methoden:
    createVoucher(payload: LexofficeVoucher): Promise<{ id: string }>
    uploadVoucherFile(voucherId: string, fileBytes: Buffer, fileName: string): Promise<void>
    getCategories(): Promise<LexofficeCategory[]>
    testConnection(): Promise<{ ok: boolean; organizationName?: string }>

auth.ts:
- getApiKey(customerId: string): Promise<string>
  → liest aus customer_credentials Tabelle WHERE customer_id = ? AND kind = 'lexoffice_api_key'
  → entschlüsselt mit pgcrypto (oder Backend-eigenem decrypt-Util — schaue wie andere Credentials gespeichert sind)

voucher.builder.ts muss exakt die Funktion buildLexofficeVoucher aus dem Konzept M05 §8 implementieren:
- type immer 'expense' für eingehende Rechnungen
- voucherNumber aus extracted_data.invoice_number
- voucherDate aus extracted_data.invoice_date (Format YYYY-MM-DD)
- totalGrossAmount aus extracted_data.total_amount
- totalTaxAmount aus extracted_data.tax_amount
- taxType: 'gross'
- voucherItems[0].categoryId: aus category.mapper.ts ermitteln
- memo: "ProzessPilot Receipt {receipt_id}"
- useCollectiveContact: true (Sammel-Kreditor — Kontaktauflösung ist Phase 3)

category.mapper.ts:
- mapSkrToLexofficeCategoryId(skrKonto: string, tenantId: string): Promise<string>
  → liest aus lexoffice_category_map Tabelle
  → Fallback: hardcoded default Lexoffice-Kategorie-ID für 'Sonstiges'
- syncCategoriesFromLexoffice(tenantId: string, customerId: string): Promise<void>
  → GET /v1/categories von Lexoffice API
  → Schreibt in lexoffice_category_map (upsert)

Typen (erstelle backend/src/core/adapters/booking/lexoffice/types.ts):
  interface LexofficeVoucher {
    type: 'expense';
    voucherNumber?: string;
    voucherDate: string;
    totalGrossAmount: number;
    totalTaxAmount: number;
    taxType: 'gross' | 'net';
    useCollectiveContact: boolean;
    contactId?: string;
    voucherItems: LexofficeVoucherItem[];
    memo?: string;
  }
  interface LexofficeVoucherItem {
    amount: number;
    taxAmount: number;
    taxRatePercent: number;
    categoryId: string;
  }
  interface LexofficeCategory {
    id: string;
    name: string;
    type: string;
  }

routes.ts — registriere folgende Endpoints (alle mit HMAC-Auth-Middleware):
  POST /api/v1/receipts/:receiptId/exports/lexoffice
    → push.handler.ts aufrufen
    → Response: { ok: true, external_id, external_url, pushed_at }

  GET /api/v1/customers/:customerId/exports/lexoffice
    → Liste alle Lexoffice-Exporte dieses Kunden (aus lexoffice_exports Tabelle)

  POST /api/v1/integrations/lexoffice/test
    → Body: { customerId }
    → Ruft lexofficeClient.testConnection() auf
    → Response: { ok: boolean, organizationName? }

  POST /api/v1/integrations/lexoffice/sync-categories
    → Body: { customerId }
    → Ruft syncCategoriesFromLexoffice() auf

push.handler.ts muss exakt den Flow aus Konzept M05 §7 implementieren:
1. Receipt aus DB laden, Status prüfen: muss 'archived' oder 'categorized' sein
2. Idempotenz: prüfen ob bereits ein Export mit target='lexoffice' und status='pushed' existiert → wenn ja, existing zurückgeben
3. buildLexofficeVoucher() aufrufen
4. Hook before_export.lexoffice feuern (über hook-runner.ts)
5. createVoucher() aufrufen
6. PDF-Anhang: erst archive_path versuchen, dann original_path als Fallback
7. uploadVoucherFile() aufrufen
8. Receipt updaten: exports Array um neuen Eintrag ergänzen, status = 'exported'
9. Hook after_export.lexoffice feuern
10. Eintrag in lexoffice_exports Tabelle schreiben
11. Response zurückgeben

Fehlerbehandlung laut Konzept M05 §12:
- 401/403 → Operator-Alert (erstmal: Error-Log + response mit klarem Fehlercode)
- 429 → Retry mit Retry-After
- 422 → receipt status = requires_review, requires_review_reason setzen
- Duplicate → requires_review mit reason 'DUPLICATE_AT_TARGET'

tests/push.handler.test.ts:
- Erfolg: Voucher erstellt + Anhang hochgeladen + Receipt auf 'exported'
- Idempotenz: zweiter Push → kein neuer Voucher, existing zurückgegeben
- 422 von Lexoffice → receipt auf requires_review
- customer_profile hat M05 nicht aktiviert → 400 Fehler

Registriere die Routes in backend/src/app.ts.

aggregator.ts — implementiere computeMonthlyAggregates(customerId, dateFrom, dateTo):
Führe exakt die SQL-Queries aus Konzept M08 §8 aus:
  1. Totals: COUNT, SUM gross, SUM net
  2. By Category: GROUP BY category, ORDER BY gross_sum DESC
  3. Top Suppliers: GROUP BY supplier_name, LIMIT 10, ORDER BY gross_sum DESC
  4. Vergleich Vormonat: selbe Queries für (year, month-1)
  5. Anomalies: Belege > 1000 EUR, neue Lieferanten (nicht im Vormonat), fehlende Kategorisierung
  Return-Typ: MonthlyAggregates { totals, by_category[], by_supplier[], comparison, anomalies[] }

WICHTIG: Die Daten liegen in receipts.extracted_data (JSONB). Felder:
  extracted_data->>'total_amount' für Brutto
  extracted_data->>'invoice_date' für Datum
  categorization->>'category_name' für Kategorie (als JSONB-Spalte)
  Passe die Queries an das tatsächliche Schema an — lies die Migrations-Dateien!

pdf-renderer.ts:
- Nutze puppeteer oder playwright (prüfe zuerst: ist puppeteer schon installiert? → npm list puppeteer)
- Falls nicht: npm install puppeteer --save (nicht global)
- Methode: fromHtml(html: string): Promise<Buffer>
  → Startet headless browser, lädt HTML, exportiert als PDF (A4, margin 20mm)
- Fallback falls puppeteer nicht installierbar: einfaches PDF mit pdfkit (npm install pdfkit)

chart-builder.ts:
- Nutze chartjs-node-canvas (prüfe ob installiert, sonst: npm install chartjs-node-canvas)
- Methode: buildCategoryBarChart(data: CategoryData[]): Promise<string> (base64 PNG)
- Chart: horizontaler Balkendiagramm, Kategorien auf Y-Achse, Summen auf X-Achse

templates/default_de.html:
Erstelle ein sauberes HTML-Template mit:
  - Kopf: ProzessPilot Logo (Text), Kundenname, Berichtszeitraum
  - KPI-Karten: Gesamt-Brutto, Anzahl Belege, größte Einzelausgabe, Veränderung zum Vormonat
  - Balkendiagramm: img Tag mit ${categoryChartBase64}
  - Tabelle: Top 10 Lieferanten (Name, Anzahl, Summe)
  - Tabelle: Auffälligkeiten (Belege > 1000 EUR, neue Lieferanten)
  - Fußzeile: Erstellt am ${date} von ProzessPilot

build.handler.ts:
POST /api/v1/customers/:customerId/reports/monthly/build
  Body: { year: number, month: number }
  1. CustomerProfile laden, prüfen ob M08 aktiviert
  2. computeMonthlyAggregates() aufrufen
  3. Hook before_report.monthly feuern
  4. buildCategoryBarChart() für das Diagramm
  5. HTML-Template befüllen
  6. fromHtml() → PDF-Buffer
  7. PDF in MinIO/Storage speichern (objectKey: {customerId}/reports/{year}-{month}/monthly.pdf)
  8. In reports Tabelle speichern
  9. Hook after_report.monthly feuern
  Return: { report_id, period, totals, pdf_object_key }

deliver.handler.ts:
POST /api/v1/customers/:customerId/reports/:reportId/deliver
  1. Report aus DB laden
  2. CustomerProfile laden → notification-Einstellungen
  3. Delivery-Dispatcher aufrufen für jeden konfigurierten Kanal
  4. In report_deliveries Tabelle schreiben
